LawAskLLM 部署备忘（公网 49.235.100.186）

一、宿主机 Nginx（重要）
- 本机 /etc/nginx/nginx.conf 只 include conf.d/*.conf，不加载 sites-enabled。
- LawAskLLM 入口：/etc/nginx/conf.d/lawaskllm.conf → 反代到 127.0.0.1:18080（容器 law_nginx）。
- 若访问 /login 为 404：多半是其它 conf.d 配置在 80 端口抢占了 49.235.100.186；已把 quantdinger 从 80 迁到 9088。
- 原 quantdinger（Vue）现入口：http://49.235.100.186:9088/ （需相应安全组放行 9088 才从外网访问）。
- 若出现 502：表示 18080 上 law_nginx 未启动，请在本项目目录执行 docker-compose up -d --build。

二、Docker 端口（与宿主机其它项目隔离）
- law_nginx：127.0.0.1:18080 → 容器 80
- law_backend：127.0.0.1:18081 → 容器 8000（调试用；前端经宿主机 Nginx → 18080 → 容器内 /api/）
- law_mysql：127.0.0.1:13306 → 容器 3306（宿主机 mysql 客户端连 13306）

三、凭据（以你机器上 .env / 数据库为准）
- 前端登录：首启且无 superadmin 时一般为 admin / Admin@123456
- MySQL：.env 中 MYSQL_PASSWORD
- JWT：.env 的 JWT_SECRET_KEY
- LLM / DashScope：LLM_API_KEY、DASHSCOPE_API_KEY

四、持久运行
- ./scripts/lawask-up.sh
- systemd：deploy/lawaskllm.service

五、环境变量
- 容器内 MYSQL_HOST=mysql；compose 会覆盖 .env 中的 MYSQL_HOST。

六、GitHub Actions CD（PROD_SSH_KEY 填什么）
- secrets.PROD_SSH_KEY 必须是「私钥」的完整文本（含 -----BEGIN ... PRIVATE KEY----- 到 -----END ...-----），不是公钥、不是空文件。
- 在任意安全机器上生成仅用于 CI 部署的密钥对（示例）：
    ssh-keygen -t ed25519 -f lawask_github_deploy -N "" -C "lawask-github-actions"
- 把「公钥」内容追加到生产机部署用户的 ~/.ssh/authorized_keys（一行一个）：
    cat lawask_github_deploy.pub | ssh PROD_USER@PROD_HOST "mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
  或手动粘贴 lawask_github_deploy.pub 那一整行。
- 把「私钥」文件 lawask_github_deploy 的全部内容复制到 GitHub：仓库 Settings → Secrets and variables → Actions → New repository secret，名称 PROD_SSH_KEY。
- 同时配置：PROD_HOST、PROD_USER；PROD_PATH 建议设为生产机上仓库根路径（如 /data/LawAskLLM）；PROD_PORT 非 22 时再填。
- 若私钥设了口令：另建 Secret「PROD_SSH_PASSPHRASE」填口令；CD 会用 ssh-keygen 在 Runner 上去掉加密后再 ssh-add。私钥已**无口令**时不要建该 Secret（误填会导致 ssh-keygen 失败）。
- 专用部署密钥也可使用 ssh-keygen -N "" 无口令，配合仅用于部署的公钥。
