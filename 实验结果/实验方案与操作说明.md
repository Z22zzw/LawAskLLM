# 法律 LLM 对照实验：方案与操作说明

本文档供本地复现与自动化（如 OpenClaw 生成批跑脚本）使用。实验预设定义见仓库 `backend/app/experiments/design.py`；批跑脚本见 `scripts/run_experiment_batch.py`。

---

## 1. 目标与题单

- **目标**：在小样本题单上对比「完整系统」「基线」「检索数据源策略」「能力消融」的客观指标与可选大模型评委四维分数。
- **题单文件（仓库根目录）**：`实验题单_20题_小样本.md`  
  - A 类（题 1–8）：法考/概念，JEC 风格  
  - B 类（题 9–16）：刑事案情，CAIL 风格  
  - C 类（题 17–20）：边界与鲁棒性  

全程保持同一 **`legal_domain`**（或留空表示综合），不要在题干外追加提示词。

---

## 2. 实验设计总览

| 编号 | 名称 | 对照组 | 实验臂 | 单次 compare 预设数 |
|------|------|--------|--------|---------------------|
| **A** | 基线对比 | `system_full` | `baseline_llm_direct`、`baseline_rag_basic` | 3 |
| **B** | 数据源策略 | `system_full` | `strategy_auto`、`strategy_jec_only`、`strategy_cail_only` | 4 |
| **C1** | 消融：关 MMR | `system_full` | `ablation_no_mmr` | 2 |
| **C2** | 消融：关 RRF | `system_full` | `ablation_no_rrf` | 2 |
| **C3** | 消融：关证据标注 | `system_full` | `ablation_no_evidence_label` | 2 |
| **C4** | 消融：关 Agent 回退 | `system_full` | `ablation_no_agent_fallback` | 2 |

说明：

- **对照组**统一为 **`system_full`**（「对比组：当前系统完整版」，已含 `source_mode: balanced` 与 MMR/RRF/证据标注/Agent 等全量开关）。
- **B 组**未单独再跑 `strategy_balanced`（与 `system_full` 的 `source_mode` 重复，无额外信息）。
- **C 组**拆成四轮 2 臂对照，以符合 HTTP 接口「单次最多 4 个预设」的约束，并控制评委单次判分臂数。

### 2.1 为何运行前要冻结 `runtime_rag_prefs.json`

`strategy_*` 与 `ablation_*` 在矩阵里**只覆盖部分键**，其余键来自 `runtime_rag_prefs.json` 与默认合并。批跑脚本会在执行前将 prefs **写入与 `system_full` 等价的 7 项开关**，跑完后**恢复原文件**，使策略/消融实验表示「在完整系统上只改一个因素」。

与 `system_full` 对齐的冻结值（脚本内常量，与 `design.py` 中 `system_full.overrides` 一致）：

- `source_mode`: `balanced`
- `use_mmr`: `true`
- `use_rrf`: `true`
- `use_evidence_labels`: `true`
- `use_agent_default`: `true`
- `force_direct_llm`: `false`
- `enable_agent_fallback`: `true`
- `active_experiment_preset`: `system_full`

## 3. 运行规模（全量）

- 题数：20  
- 每题跑：A 一次（3 臂）+ B 一次（4 臂）+ C1～C4 各一次（各 2 臂）= **每题 6 次** `compare` 批处理  
- 全量批次数：20 × 6 = **120 次** `compare_arms_parallel` 调用  
- 若开启大模型评委：每批再调用 **1 次** `llm_score_compare_arms`（四维打分）  
- 扁平化 CSV 行数：20 × (3+4+2×4) = **300 行**（每行一个 preset 臂）

---

## 4. 数据产出与目录约定

根目录下批跑默认输出与结构如下；也可通过 `--out-dir` 重定向到本目录下子文件夹。

| 文件 | 含义 |
|------|------|
| `exp_all.csv` | 全部实验行 |
| `exp_A.csv` | 仅 A |
| `exp_B.csv` | 仅 B |
| `exp_C1.csv` … `exp_C4.csv` | 各轮消融 |
| `experiment_batch_meta.json` | 本次运行参数（时间、是否评委、题单路径等） |
| `run.log` | 若用手动 `tee` 保留的控制台日志（可选） |

**建议目录结构（本仓库已部分存在）：**

```text
实验结果/
  实验方案与操作说明.md    ← 本文档
  大模型评分/              ← `--out-dir` 指向此处：默认开启四维评委
  大模型不评分/            ← `--no-llm-score`：仅客观指标与 composite，更快
```

---

## 5. CSV 列说明（批跑脚本导出）

主要列：`exp`、`question_id`、`block`、`legal_domain`、`preset_id`、`group`、`is_control`（`system_full` 为 1）、`latency_ms`、`citation_count`、`answer_length`、`chain_trace_len`、`intent`、`skipped_retrieval`、四维 `llm_*`、`llm_avg`、`llm_note`、`llm_score_note`（整批评委说明或跳过原因）、单次对照内归一化指标 `latency_score_0_1`、`citation_score_0_1`、`trace_score_0_1`、`composite_0_1`、`rank_composite`、`question_preview`、`status`、`created_at_utc`、`prefs_frozen`。

`composite_0_1` 等归一化逻辑与线上一致，见 `backend/app/experiments/analytics.py`（质量权重约 58%，速度与引用等为辅助）。

## 6. 命令行操作（仓库根目录）

前置：**在 `backend` 目录执行**，并设置 `PYTHONPATH=.`，以便 `app` 包可导入。

```bash
cd /path/to/LawAskLLM/backend
```

### 6.1 全量 + 大模型评委（输出到本目录「大模型评分」）

```bash
PYTHONPATH=. python ../scripts/run_experiment_batch.py \
  --out-dir "../实验结果/大模型评分"
```

### 6.2 全量 + 关闭评委（更快；输出到「大模型不评分」）

```bash
PYTHONPATH=. python ../scripts/run_experiment_batch.py \
  --no-llm-score \
  --out-dir "../实验结果/大模型不评分"
```

### 6.3 调试：只跑实验 A 且前 2 题

```bash
PYTHONPATH=. python ../scripts/run_experiment_batch.py --only A --max-questions 2
```

### 6.4 固定法律领域

```bash
PYTHONPATH=. python ../scripts/run_experiment_batch.py \
  --legal-domain "刑法" \
  --out-dir "../实验结果/大模型评分"
```

### 6.5 环境变量（大模型评委与 RAG 生成）

- 评委与 RAG 共用 **`LLM_API_KEY`**（见 `backend/app/core/config.py`）。  
- 脚本会在导入应用前加载仓库根目录 **`.env`**（若已安装 `python-dotenv`）；跑完后请在 `experiment_batch_meta.json` 中确认 `llm_score` 与 `llm_score_note` 是否仍为「未配置已跳过」。  
- 嵌入/向量若单独配置，见 `DASHSCOPE_API_KEY` 等，与题述实验并行无关。

---

## 7. 与 HTTP API 的对应关系（可选）

- `POST /api/v1/experiments/compare`  
- Body：`question`、`legal_domain`、`preset_ids`（2～4 个）、`llm_score`  
- 返回：`arms`、`analysis`（与 CSV 中归一化列同源）  

批跑脚本等价于多次调用该接口的逻辑，但不写数据库用户历史表（**ExperimentCompareRun**），仅写 CSV。

---

## 8. 给 OpenClaw / 自动化生成脚本的提示

可将下列要点作为生成任务时的约束：

1. **工作目录**：`LawAskLLM/backend`，环境变量 `PYTHONPATH=.`
2. **唯一入口脚本**：`../scripts/run_experiment_batch.py`
3. **全量任务**：无 `--only`、无 `--max-questions`；评委版与不评分版各跑一遍，输出到 `实验结果/大模型评分` 与 `实验结果/大模型不评分`
4. **失败重试**：脚本单题失败会写 `status=failed:...`；自动化可对失败 `question_id`+`exp` 重跑
5. **耗时**：全量 120 批 + 评委 API 可能达数十分钟级，宜设置合理超时与日志 `tee`
6. **prefs**：脚本自动备份/恢复 `runtime_rag_prefs.json`，无需手工改文件；避免并行跑两个实例互相覆盖
